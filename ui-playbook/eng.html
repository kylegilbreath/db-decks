Databricks Design · 2026

The current state

  • Figma specs die in implementation.
  • UI issues ship because no one owns the design review.
  • Non-engineer PRs create review burden with no clear process.
Here's what we're trying.

A system that lets non-engineers engage, prototype, and create using plans and living specs — while holding engineers accountable for design, too.

Non-engineers
Provide a functional spec — a code PoC or a plan.md.
Engineers
Provide technical stability and design accountability.

The four paths

Path Best for Your job Outcome
Direct Fix Nits, text, CSS Safety review, fast merge Production
Plan-Build New features Sign off plan.md first Production
Vibe POC Research, "what if?" Review behavior, not code Living spec → rebuild
Figma High-fidelity UI Implement to spec, no improvising. Production

You'll interact with all four. Your role changes depending on which path the non-engineer chose.

Two jobs

You already do this
Technical stability
Code quality. CI. Architecture review. Catching regressions before merge.
This is the new ask
Design accountability
You're often the last person who sees the UI before it ships. If something looks wrong, say something.

You’re not here to block shipping. If scope creeps, nudge toward a plan — designers included.
“This is getting big — mind a plan.md so we can break it into steps?”

When to loop in a designer

Trigger Action
Touches a core user flow (auth, nav, data creation/deletion) Tag designer in PR
Introduces a new UI pattern Tag designer in PR
Change spans more than one surface or component Tag designer in PR
Spec feels ambiguous or missing Slack designer before building

The goal isn't more process. It's catching the right things early instead of fixing them after merge.

Before you merge.
  • No UI PR without a visual receipt — screenshot or screen recording, no exceptions.
  • If you notice something off in the UI, say something. Even a comment counts.
